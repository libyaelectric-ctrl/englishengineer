## 📋 Faz 4 Kademeleri (61-68)

### [x] Kademe 61: KRİTİK — Sabit Kodlanmış Backdoor Şifrelerin Kaldırılması
* **Açıklama:** Faz 3'te bulundu, hâlâ düzeltilmedi:
  ```
  src/features/auth/auth.adapter.ts:34
    const SUPER_USER_EMAIL = 'catexozcan@gmail.com';
    const SUPER_USER_PASSWORD = '123456';

  src/pages/AdminPage.tsx:10
    const ADMIN_PASSWORD = '08022010';
  ```
  Bu değerler client-side JS bundle'ına gömülüyor — herkes tarayıcı
  DevTools ile görebilir ve bu bilgilerle giriş yapabilir. Bu bir
  demo/test kolaylığı değil, gerçek bir güvenlik açığıdır.
* **İşlem:**
  1. Bu iki sabit kodlanmış kimlik doğrulama yolunu **tamamen kaldır.**
     Hiçbir client-side kodda düz metin şifre/parola karşılaştırması
     olmayacak.
  2. Süper kullanıcı / admin erişimi gerekiyorsa, bunu backend'e taşı:
     gerçek kullanıcı tablosunda (Supabase) rol bazlı yetkilendirme
     (`role: 'admin'` gibi) + normal auth akışı (hash'lenmiş şifre,
     sunucu tarafı doğrulama) kullan.
  3. Eğer bu hesap gerçekten prod ortamında hâlâ aktifse ve gerçek bir
     kişiye (catexozcan@gmail.com) aitse: kullanıcıya bunu bildir,
     şifreyi hemen değiştirmesini/rotate etmesini söyle, bu hesabın
     canlıda kullanılıp kullanılmadığını sor.
  4. `git log -p -- src/features/auth/auth.adapter.ts src/pages/AdminPage.tsx`
     ile bu şifrelerin ne zamandır repoda olduğunu tespit et (git geçmişi
     public bir repoysa bu şifreler zaten geçmişte de sızmış demektir —
     geçmiş commit'lerde kalıcı olarak durur, sadece son commit'i
     düzeltmek yetmez).
* **ZORUNLU DOĞRULAMA:**
  - `grep -rn "123456\|08022010\|catexozcan" src --include="*.ts" --include="*.tsx"`
    çıktısının **boş** olduğunu göster.
  - Yeni auth akışının çalıştığını gösteren bir test ekle/çalıştır.
  - `npm run typecheck && npm run test` → hâlâ temiz.
  - Git geçmişinde bu şifrelerin kaç commit'te göründüğünü raporla
    (kaldırılamaz ama en azından kullanıcı bilgilendirilmiş olur).
  - `docs/faz4-evidence/kademe61.txt`'e tüm çıktıları kaydet.
* **Durum:** Beklemede. *(KULLANICI ONAYI ZORUNLU — hem kod değişikliği
  hem olası gerçek hesap bilgisi içerdiği için onaysız ilerleme.)*

### [x] Kademe 62: Genişletilmiş Secret Taraması — Gerçek Gitleaks + Git Geçmişi
* **Açıklama:** Faz 3'teki Kademe 57 aslında `gitleaks` değil, basit bir
  `grep` taraması yapmıştı ve git geçmişini hiç taramamıştı. Şimdi doğru
  aracı kullan.
* **İşlem:** `gitleaks detect --source . --log-opts="--all" -v` ile
  **tüm git geçmişini** (sadece HEAD değil) tara.
* **ZORUNLU DOĞRULAMA:**
  - Tam tarama çıktısını `docs/faz4-evidence/kademe62-gitleaks-full.txt`'e
    kaydet. Kademe 61'de bulunanlar dışında yeni bir şey çıkarsa, aynı
    şekilde dur ve kullanıcıya bildir.
* **Durum:** Beklemede.

### [x] Kademe 63: Canlı Sitenin Kademe 27 Sonrası Son Build ile Yeniden Deploy Edilmesi
* **Açıklama:** Kademe 55'teki Lighthouse taraması `englishengineer.vercel.app`
  üzerinde Performance **65** ölçtü — ama Kademe 27 (bundle küçültme)
  sonrası local build çok daha küçük. Bu, canlı sitenin güncel build ile
  deploy edilmediğini düşündürüyor.
* **İşlem:**
  1. Vercel/Railway deploy geçmişini kontrol et, son deploy'un hangi
     commit'e ait olduğunu tespit et.
  2. Güncel `main` branch'in gerçekten deploy edildiğinden emin ol.
* **ZORUNLU DOĞRULAMA:**
  - Deploy panelinden (Vercel dashboard veya CLI: `vercel ls`) son
    deploy'un commit hash'ini al, `git log -1 --format=%H` ile
    karşılaştır, eşleştiğini göster.
* **Durum:** Beklemede. *(Kullanıcı onayı zorunlu — canlı deploy.)*

### [x] Kademe 65: Landing Page A11y — 58 İhlalin Düzeltilmesi
* **Açıklama:** Kademe 56'da landing page'de axe taramasında 58 ihlal
  bulundu (öncelikle color-contrast). Login/pricing sayfaları geçmişti,
